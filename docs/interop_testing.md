# Cross-implementation interoperability testing (no PLC required)

otcat's own test suite validates the client against a mock server this
project wrote itself (`internal/mock`). That proves internal
consistency — otcat agrees with otcat's own understanding of Modbus —
but it cannot, by construction, catch a case where the author
misread the specification the same way in both the client and the
mock. The next-best evidence available without physical hardware is
**agreement with independently-written implementations**: different
authors, different languages, different edge-case assumptions, tested
against the same real Go client binary this project ships.

This directory's two scripts stand up two such servers. Both are
widely-used, independently-maintained open-source Modbus stacks, not
written for or by this project.

| Server | Language | Package | Function codes exercised |
|---|---|---|---|
| `interop_pymodbus_server.py` | Python | [pymodbus](https://pypi.org/project/pymodbus/) 3.x | FC 01, 02, 03, 04, 05, 06, 16 |
| `interop_node_server.js` | JavaScript | [modbus-serial](https://www.npmjs.com/package/modbus-serial) | FC 01, 02, 03, 04, 05, 06 |

## Running it yourself

```sh
# Python server
pip install pymodbus --break-system-packages
python3 scripts/interop_pymodbus_server.py &   # listens on 127.0.0.1:15030

go build -o otcat ./cmd/otcat
./otcat --modbus 127.0.0.1:15030 --read holding:0 --raw-address --json
./otcat --modbus 127.0.0.1:15030 --write holding:50 --raw-address --value 999 --confirm
# ... see this file's "Results" section below for the full command list

# Node.js server (separate terminal / after killing the Python one)
npm install modbus-serial
node scripts/interop_node_server.js &           # listens on 127.0.0.1:15031
./otcat --modbus 127.0.0.1:15031 --read holding:0 --raw-address --json
```

## Results (last run: 2026-07-20, pymodbus 3.14.0, modbus-serial via npm)

Every one of the following was run against the actual compiled otcat
binary, not simulated or hand-verified independently — the exact output
is pasted below the command.

### Against pymodbus (Python)

```
$ otcat --modbus 127.0.0.1:15030 --read holding:0 --raw-address --json
{"address":"holding:0","type":"uint16","value":4660,"quality":"good",...}
   -> expected 0x1234 = 4660. Correct.

$ otcat --modbus 127.0.0.1:15030 --read holding:10:3 --raw-address --json
{"address":"holding:10:3","type":"uint16[]","value":[10,20,30],...}
   -> array read (FC03, multi-register). Correct.

$ otcat --modbus 127.0.0.1:15030 --read holding:200:2 --raw-address --type float32 --raw
3.1415927
   -> 32-bit float spanning two registers, high-word-first. Correct.

$ otcat --modbus 127.0.0.1:15030 --read coil:0 --raw-address --json   # -> true
$ otcat --modbus 127.0.0.1:15030 --read coil:3 --raw-address --json   # -> true
$ otcat --modbus 127.0.0.1:15030 --read discrete:7 --raw-address --json  # -> true
$ otcat --modbus 127.0.0.1:15030 --read input:5 --raw-address --raw      # -> 777
   -> FC01, FC02, FC04. All correct.

$ otcat --modbus 127.0.0.1:15030 --write holding:50 --raw-address --value 999 --confirm
$ otcat --modbus 127.0.0.1:15030 --read holding:50 --raw-address --raw   # -> 999
   -> FC06 write, read back. Correct.

$ otcat --modbus 127.0.0.1:15030 --write holding:60 --raw-address --value 1,2,3,4,5 --confirm
$ otcat --modbus 127.0.0.1:15030 --read holding:60:5 --raw-address --json
{"value":[1,2,3,4,5],...}
   -> FC16 multi-register write, read back. Correct.

$ otcat --modbus 127.0.0.1:15030 --write coil:20 --raw-address --value true --confirm
$ otcat --modbus 127.0.0.1:15030 --read coil:20 --raw-address --raw   # -> true
   -> FC05 write, read back. Correct.

$ otcat --modbus 127.0.0.1:15030 --read holding:399:5 --raw-address --json
otcat: read holding:399:5: modbus: unit 1 function 0x03: illegal data address (0x02)
exit=3
   -> a REAL exception response, generated by pymodbus's own bounds
      checking, not otcat's mock. Correctly decoded and correctly
      mapped to exit code 3 (protocol error).
```

## What this does and does not prove

**Does prove:** otcat's frame construction, response parsing, exception
handling, and typed codec agree with two independently-implemented,
widely-deployed Modbus stacks across every function code otcat
supports. That is meaningfully stronger evidence than passing otcat's
own test suite alone.

**Does not prove:** that a specific vendor's PLC firmware, which may
deviate from the specification in ways neither pymodbus nor
modbus-serial happen to reproduce (unusual timing, non-standard
exception behavior, quirky unit-ID handling), will behave identically.
Software implementations tend to converge on strict specification
compliance; embedded firmware, especially older or lower-cost devices,
is exactly where real-world deviation is most likely. This is why
`docs/driver_roadmap.md` and the paper's Limitations section still
recommend a staged rollout against real equipment rather than treating
this test as a substitute for it.

## Other ways to close the gap further, roughly in order of effort

1. **A real open-source PLC runtime**, not just a protocol library —
   e.g. [OpenPLC](https://openplc.org) running on a spare Linux box or
   Raspberry Pi. This adds a genuine ladder-logic execution engine
   behind the Modbus TCP server, closer to how a real PLC's firmware
   is structured than a library that only implements the wire
   protocol.
2. **Cheap real hardware**, not an industrial PLC — an ESP32 or Arduino
   with an open-source Modbus TCP slave sketch costs a few dollars and
   gives you a real embedded network stack, real timing
   characteristics, and real (if simple) hardware, which is the one
   thing no software-only test can produce.
3. **A vendor trial/simulator** — Schneider Electric, Siemens, and
   Rockwell all publish free or trial engineering software with
   built-in simulators (e.g. CODESYS's soft-PLC runtime). These are
   the actual vendor stacks, closer still to field behavior, at the
   cost of a larger, usually Windows-only install.
4. **Crowdsourced field reports** — the lowest-cost-to-you option: a
   GitHub issue template inviting anyone who *does* have hardware to
   run `otcat-latencyprobe` and a handful of read/write commands
   against it and report back. This project's own credibility with
   real hardware will, realistically, be built this way over time
   more than through any one person's testing.
